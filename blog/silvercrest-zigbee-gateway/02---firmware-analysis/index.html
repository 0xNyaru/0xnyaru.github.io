[Silvercrest Smart Home] Firmware analysis

Introduction to hardware hacking

1. Introduction#

In the previous section, the firmware was successfully extracted using various methods. Now, it’s time to analyze the firmware to identify the root user’s password.

2. Firmware Analysis#

This phase focuses on recovering the root password. The goal is to simplify access to the gateway once powered on — particularly using SSH, which is significantly faster than a UART connection at 38400 baud.

This also avoids the need to emulate the firmware prematurely.

After some exploration, information related to the root password appears to reside within a program located in the jffs2-fs partition, at the following path:

jffs2-root/tuyamtd
plaintext

To recall, you can extract the partition contents using binwalk:

sudo docker run -t -v "$PWD":/analysis binwalkv3 -Me jffs2-fs.bin
sh

A detailed breakdown of the filesystem and all programs will be covered in a separate post. Here, the sole objective is to recover the root password.

Two binaries are particularly important in this step:

  • tyZ3Gw
  • tuyamtd

The analysis is done using Binary Ninja.

2.1 Analysis of tyZ3Gw#

Inside the main function, the tyZ3Gw binary reveals a hardcoded UUID and authentication key:

Later in the code, the function tuya_login_init is called with g_auth_key as a parameter. This function shows the expected user account is clearly root:

A little further on, the logic behind defining the password is observed:

The program extracts the last 8 characters of the g_auth_key and uses them as the password. However, this does not allow successful login to the system — suggesting that the current value of g_auth_key is incorrect or incomplete.

Back in the main function, another function draws attention: get_prod_info. It fills the g_auth_key variable using the value of auzkey, by calling another binary — tuyamtd.

This auzkey becomes critical in order to retrieve the root access:

Summary:

  • The program tuyamtd is executed with:
    • read
    • auzkey
  • The value of auzkey initializes g_auth_key
  • The root password is the last 8 characters of g_auth_key

2.2 Analysis of tuyamtd#

After static analysis, the content of the main function can be inspected. After some renaming, the following structure becomes visible:

This screenshot reveals that:

  • The program reads from /dev/mtd3 (partition tuya-label)
  • It takes two CLI arguments
    • read or write
    • a key (ex: auzkey)

TIP

What is /dev/mtd3? It refers to a NOR/NAND flash memory partition, exposed by the Linux kernel’s MTD subsystem.. This type of partition often contains:

  • firmware
  • configuration data
  • crypto keys
  • credentials (root/admin)
  • persistent logs

The flash_read function then calls do_read_encrypt(int matched, char *arg2):

Here, a call to lseek moves the read pointer in flash memory. The address is calculated as:

1 * 0x1c + 0x16010 -> 0x2000
plaintext

The function reads 98 bytes, from 0x2000 to 0x2062. The data can be viewed directly in the partition dump:

xxd tuya-label.bin|grep grep 2000 -A 3

00002010: d656 ed41 b658 65cc 4090 ec9a 604b f68a  .V.A.Xe.@...`K..
00002020: a59a 0b67 3a9e 0e1c d107 5403 948a ac14  ...g:.....T.....
00002030: 97ae ffff ffff ffff ffff ffff ffff ffff  ................
sh

The function then decrypts the data using AES-128 in ECB mode, with a key stored in g_encode_key:

The next step is to locate g_encode_key. One trick is to invoke tuyamtd with encode_key instead of auzkey, which causes matched to be 0.

The address becomes:

0 * 0x1c + 0x16010` -> `0x1800`
plaintext

Extract the memory segment:

xxd tuya-label.bin|grep 01800 -A 1

00001800: 1010 7236 2730 2d2a 3969 3f30 6f72 576c  ..r6'0-*9i?0orWl
00001810: 2778 ffff ffff ffff ffff ffff ffff ffff  'x..............
sh

The AES key is now accessible.

To use it, we must understand how the key is encoded. The function responsible is encode_key_get:

The logic can be reimplemented like this:

from Crypto.Cipher import AES
import struct

encoded_key = b"r6'0-*9i?0orWl'x"
decoded_key = b""
for i in encoded_key:
    tmp = (encoded_key[0] * i % 0x5D) + 0x21
    decoded_key += chr(tmp).encode()

enc_auzkey = bytes.fromhex(
    "E12E1D3F9EDEB6F89F7D1CBA65CDD656ED41B65865CC4090EC9A604BF68AA59A"
)

c = AES.new(decoded_key, AES.MODE_ECB)
dec_auzkey = c.decrypt(enc_auzkey)

print(f"decoded key: {decoded_key}")
print(f"auzkey: {dec_auzkey}")
print(f"root password {dec_auzkey[-8:]}")
python

3. Conclusion#

This tutorial walked through the reverse engineering of a Silvercrest IoT gateway sold by Lidl, focusing on firmware extraction and analysis.

The process began with identifying debug interfaces like UART, allowing direct interaction with the bootloader. Serial logs revealed key hardware and memory layout information.

Firmware extraction was done via UART, and tools like pyserial and binwalk were used to analyze partitions

The main goal — gaining root access — was achieved through static and dynamic analysis of two binaries: tyZ3Gw and tuyamtd. The root password was recovered by reverse-engineering the AES-based encryption scheme used to store the key in flash memory.

This analysis highlights the critical importance of proper authentication, encryption, and secure storage of secrets in embedded systems.

That’s it for this series — at least for now. There doesn’t seem to be much value in further analyzing:

  • the mobile application, as it’s just a Tuya-branded app (yes, even the Lidl one)
  • other binaries on the device, as the gateway seems outdated and identifying a 0-day would likely be time-consuming with limited impact
  • the Zigbee communication: some strings in the firmware suggest the use of the default key, but exploiting this would require intercepting traffic to retrieve the Network Key, and I didn’t have the required hardware at the time of writing
